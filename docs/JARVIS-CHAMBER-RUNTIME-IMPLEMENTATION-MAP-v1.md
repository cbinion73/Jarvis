# JARVIS Chamber Runtime Implementation Map v1

## Purpose

This document turns the chamber spec and stewardship-lanes spec into a concrete implementation map for the current JARVIS repo.

It answers four questions for each surface:

1. Which runtime producer already exists?
2. Which service endpoint already exposes it?
3. Where does the current shell touch it?
4. Should we reuse it, reshape it, or hide it?

This is intentionally grounded in the real repo surface:

- [jarvis/runtime.py](/Users/chris/Desktop/CODE/JARVIS/jarvis/runtime.py)
- [jarvis/service.py](/Users/chris/Desktop/CODE/JARVIS/jarvis/service.py)
- [jarvis/voice_ui.py](/Users/chris/Desktop/CODE/JARVIS/jarvis/voice_ui.py)

## Core Mapping Rule

The chamber should not invent a new backend if the current runtime already produces the right truth in a fragmented form.

The primary implementation move is:

- consolidate existing producers into chamber-native payloads
- simplify the shell around those chamber payloads
- demote or hide packet-era machinery that breaks the chamber feeling

## Current Surface Inventory

The current repo already has strong raw material:

- `dashboard_snapshot()` for shell-wide summary aggregation
- `today_board()` for priorities, carry, and assistant notifications
- `morning_brief()` and `first_light_check()` for spoken briefing posture
- `cadence_review()` and `proactive_state_snapshot()` for drift and interruption posture
- `chat_state_snapshot()` and conversation APIs for the relationship surface
- `list_pending_approvals()` and `work_lifecycle_snapshot()` for decisions and staged work
- `self_improvement_snapshot()` and `guardian_status_snapshot()` for chamber operations

The problem is not missing data.

The problem is that the current shell still presents those truths through:

- packet navigation
- floating panel logic
- exposed scene and packet machinery
- multiple overlapping home concepts

## Chamber Zone Mapping

### 1. Briefing

Desired chamber meaning:

- what matters now
- what JARVIS noticed
- what JARVIS prepared
- what needs a decision
- what JARVIS recommends next

Current runtime producers:

- `morning_brief(actor_name)`
- `first_light_check(actor, device_id, force, timezone_name)`
- `today_board(actor_name)`
- `dashboard_snapshot(actor_name)`
- `daily_strategic_brief(actor_name)`
- `cross_domain_synthesis_brief(actor_name, topic)`

Current service endpoints:

- `GET /api/briefing`
- `GET /api/first-light`
- `GET /api/today-board`
- `GET /api/dashboard`
- `GET /api/strategic-brief`
- `GET /api/cross-domain-brief`

Current UI touchpoints:

- `render_voice_shell(...)` bootstraps a shell with `briefing` as an allowed packet
- `openPacket("briefing")`
- `packetFromRequest(...)` routes status and agenda-style prompts toward briefing or today packets
- first-light and briefing loaders in [jarvis/voice_ui.py](/Users/chris/Desktop/CODE/JARVIS/jarvis/voice_ui.py)
- scene shell header and summary in the current center surface

Implementation direction:

- reuse `today_board()` and `first_light_check()` as the chamber briefing foundation
- add a new chamber-native runtime producer, likely `chamber_home_snapshot()`, that merges:
  - first-light narrative opening
  - today-board priorities
  - one recommended next move
  - one prepared-work summary
  - one decision count summary
- stop treating briefing as just another packet
- promote briefing to the default visible home surface in [jarvis/voice_ui.py](/Users/chris/Desktop/CODE/JARVIS/jarvis/voice_ui.py)

Status:

- producer exists in pieces
- needs consolidation

### 2. Already Working

Desired chamber meaning:

- visible proof that JARVIS has been quietly preparing useful work

Current runtime producers:

- `assistant_notifications(...)`
- `work_lifecycle_snapshot(actor_name, limit)`
- `self_improvement_snapshot()`
- `dashboard_snapshot(actor_name)["assistant_surface"]`
- active lifecycle and staged work generated by Catalyst and maintenance flows

Current service endpoints:

- `GET /api/assistant-core/notifications`
- `GET /api/work-lifecycle`
- `GET /api/self-improvement`
- `GET /api/dashboard`

Current UI touchpoints:

- dashboard and today packets rendering assistant inbox items
- lifecycle inspector packet
- self-improvement packet and sandbox status views
- assorted `core-home-item` blocks in [jarvis/voice_ui.py](/Users/chris/Desktop/CODE/JARVIS/jarvis/voice_ui.py)

Implementation direction:

- reuse `work_lifecycle_snapshot()` as the main source of meaningful prepared work
- supplement with `self_improvement_snapshot().active_runs` only when the work is meaningful enough to mention
- do not surface raw queue depth, raw agent activity, or low-signal maintenance chatter
- in the chamber, this zone should show outcome phrasing such as:
  - preparing tomorrow's meeting frame
  - staging a diligence brief
  - reviewing family schedule friction

Status:

- producers exist
- current shell over-exposes operational texture
- needs a calmer projection layer

- `GET /api/today-board`

- `POST /api/work-lifecycle/{work_id}/action`

- phrase every item as a verb-led decision, not a system state

Status:

- strong producer and endpoint coverage already exists
- mostly a presentation and prioritization problem

## Stewardship Lane Mapping

### 1. Family Stewardship Lane

Lane purpose:

- protect household peace
- detect family scheduling friction
- keep family rhythm visible without turning the family into a dashboard

Current runtime producers:

- `family_mode_brief(mode)`
- `family_calendar_summary()`
- `merged_calendar_events(limit)`
- `merged_calendar_brief(limit)`
- `today_board(actor_name)["calendar"]`
- family planning and household support handlers in runtime

Current service endpoints:

- `GET /api/family-calendar`
- `GET /api/merged-calendar`
- legacy family workflow handlers via `/api/family-plan`, `/api/rebekah-center`, `/api/grocery-support`, `/api/meal-plan`, `/api/vehicle-plan`

Current UI touchpoints:

- family scene preset
- briefing packet
- today packet
- household-oriented packets

Implementation direction:

- feed family lane output into:
  - the primary briefing
  - `Needs You` when a family decision is blocked
  - `Drift / Risk` when dinner, rest, or logistics are threatened
- keep grocery, meal, and vehicle mechanics secondary and on-demand

## UI File Mapping

### [jarvis/voice_ui.py](/Users/chris/Desktop/CODE/JARVIS/jarvis/voice_ui.py)

Current role:

- renders the whole shell
- contains the packet strip, scene shell, center core, chat panel, and many packet renderers

Current chamber blockers:

- packet strip is still a primary navigation model
- scene and packet concepts are more visible than the chamber doctrine allows
- floating panel and layout-edit capabilities reinforce “tool shell” over “intelligence chamber”
- the chat surface feels like a side panel more than a relationship lane

Primary implementation move:

- refactor the shell around a dedicated home chamber layout
- keep packet infrastructure temporarily for secondary detail surfaces
- make the home render path separate from packet-era chrome

Recommended file-level work:

1. Add a new chamber home renderer that draws the five zones directly.
2. Keep packet rendering for secondary drill-in views only.
3. Collapse packet-strip visibility on default home load.
4. Reposition or redesign `.chat-window` and `.chat-composer` to feel chamber-native.
5. Route dashboard refresh into chamber-zone hydration instead of packet-first hydration.

## Success Condition

This mapping is working when:

- the home surface is powered by one chamber-native aggregate payload
- the home surface feels like one coherent briefing, not packet hopping
- the five chamber zones each have a real runtime producer
- the stewardship lanes contribute visible outcomes without exposing swarm machinery
- the packet-era shell remains available for depth, but no longer defines the first impression
