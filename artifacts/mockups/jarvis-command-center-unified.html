Unified Frontend System

JARVIS should feel like a briefing-led command center, not a dashboard or a chat shell.

This artifact shows one shared design language across desktop and iPhone: posture strip, dominant briefing field, decision lane, quiet stewardship, contained drift visibility, and a stable command dock with optional voice. The goal is oversight, continuity, prioritization, review, triage, and one-tap command.

Shared Posture 1 shell

Both platforms inherit the same hierarchy and vocabulary.

Command Model Silent-first

Type, tap, approve, defer, redirect, then use voice when it helps.

Navigation 5 roots

Small stable root navigation with grouped workstreams underneath.

Continuity Always near

Command and prior context stay close without taking over the screen.

Shared Rules

  • Briefing before browsing. The first read should explain reality before exposing deeper tools.
  • Voice-enabled, not voice-first. The shell must work beautifully when the user never speaks.
  • Command over exploration. Navigation is for steering an active system, not wandering feature silos.
  • Prepared action beats blank composition. JARVIS should more often stage good choices than ask the user to start from zero.
  • One coherent presence. Agent society stays mostly hidden until visibility improves trust or review.

Platform Split

Desktop

Parallel visibility. Briefing, decisions, and command can coexist in one steady frame.

iPhone

Thumb-reachable orientation. One dominant narrative with compressed support lanes and one-tap decisions.

Shared

Same vocabulary, same color meanings, same recommendation framing, same command semantics.

Never

No long root tab strip of equal-weight domains, no blank chat-first home, no generic KPI dashboard wall.

Future mockups and implementation should keep these invariants: one dominant briefing field, small stable root navigation, command always near, silent-first completion paths, and workstream depth grouped beneath a shared shell.
