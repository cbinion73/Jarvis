# JARVIS Arrival And Conversation Workspace Doctrine

## Purpose

This document defines how it should feel to enter JARVIS.

JARVIS does not have a Home screen.

JARVIS does not launch into a dashboard.

JARVIS does not open an application.

JARVIS is a continuously operating system.

Chris does not launch JARVIS.

Chris rejoins JARVIS.

The purpose of the arrival experience is not to display information.

The purpose is to restore continuity.

Read this after:

- `docs/JARVIS-LIFE-OPERATING-OFFICER.md`
- `docs/jarvis-canonical-operating-model.md`
- `docs/JARVIS-SESSION-STATE.md`

If this document conflicts with the Life Operating Officer directive, the Life
Operating Officer directive wins.

## Canonical Frame

The user should never feel:

`I opened software.`

The user should feel:

`I rejoined an operation that has been running in my absence.`

This is the defining emotional characteristic of JARVIS entry.

JARVIS is not waiting in an idle state for Chris to provide a command.

JARVIS should already be paying attention, advancing missions, maintaining
watches, and monitoring what matters.

When Chris arrives, JARVIS should hand back situational awareness.

That handoff is a ritual.

It is not a report.

It is not a dashboard.

It is not a notification pile.

## Laws

### Law 4

JARVIS begins every interaction by restoring continuity.

Before answering questions.

Before opening workspaces.

Before displaying missions.

Before presenting recommendations.

JARVIS first reconnects Chris with the current state of his life.

### Law 5

The permanent layer should communicate presence, continuity, and readiness -
not information.

Information belongs to dynamic workspaces.

Presence belongs to the operating layer.

### Law 6

Every transient layer should disappear once its purpose is fulfilled.

The handoff exists to restore continuity.

Once continuity is restored, the handoff should disappear.

The workspace exists to support a mission, question, decision, or action.

Once the context changes, the workspace should transform or disappear.

Nothing should linger simply because it was previously displayed.

## The Permanent Operating Layer

The permanent operating layer always exists.

It is not a screen.

It is not a workspace.

It is the operating substrate of JARVIS.

The permanent layer should remain minimal, calm, and alive.

If it becomes information-dense, it has drifted toward Home.

### 1. Conversation Presence

Conversation Presence communicates:

`JARVIS is here.`

Capabilities include:

- active conversation state
- voice interaction
- text interaction
- interruption support

The user should always feel that conversation is immediately available.

This should feel like presence, not a chat box.

### 2. Continuity Spine

Continuity Spine communicates:

`Life has continued while you were away.`

Examples may include:

- active missions
- changes detected
- recommendations available
- watches running

The Continuity Spine is not a dashboard.

It is a pulse.

It should reassure rather than compete for attention.

### 3. Ambient Intelligence

Ambient Intelligence communicates:

`JARVIS is paying attention.`

Examples may include:

- monitoring
- watches
- autonomous activity
- active intelligence gathering

The purpose is trust.

Not reporting.

### 4. Mission Gravity

Mission Gravity communicates:

`These are the things currently exerting force on your life.`

Examples may include:

- Summer Camp
- Health
- Book Launch
- JARVIS

Gravity is not the same thing as priority.

Gravity is significance.

It is the set of missions, concerns, or commitments most likely to shape what
matters right now.

## The Operational Handoff

The Operational Handoff is a ritual.

It is not content.

It is not a report.

It is not a dashboard.

It is a transfer of situational awareness from the always-on system to Chris.

The handoff should feel like someone who has been watching the shop while Chris
was gone is now returning the keys.

### Opening Sequence

1. Reconnect
2. Orient
3. Prioritize
4. Recommend
5. Converse

### Example

Good afternoon Chris.

While you were away:

- Summer Camp advanced
- Book sales were monitored
- Workshop inventory changed
- overnight JARVIS development was reviewed

One mission needs attention.

One opportunity deserves review.

I recommend looking at your health mission today.

What would you like to do?

### Handoff Rules

The handoff:

- unfolds over time
- is narratively sequenced
- is interruptible at any moment
- terminates immediately when conversation takes priority

If Chris begins speaking, the handoff stops.

The conversation becomes primary.

The system should not insist on finishing the ritual if the human has already
moved on.

## Dynamic Workspace Doctrine

Workspaces are not permanent.

Workspaces emerge from intent.

Examples:

- `Am I ready for summer camp?` -> Summer Camp workspace appears
- `How are book sales?` -> Publishing workspace appears
- `How am I doing on health goals?` -> Health workspace appears

The workspace is generated from:

`Object + Posture + Context`

The workspace should feel materialized rather than navigated to.

It should exist because the current conversation requires it, not because it is
part of the default shell.

### Workspace Materialization Rules

A workspace should appear only when it is:

- requested
- inferred from intent
- required to support a recommendation
- required to support a decision

Workspaces should not exist simply because they are available.

### Workspace Transition Rules

The first dynamic layer after re-entry should be the Operational Handoff Stack.

It exists only to restore continuity.

Once the handoff has done its job, it should disappear.

When Chris expresses intent, the handoff should collapse and the relevant
workspace should materialize.

This transition should feel like moving from briefing into work.

## Anti-Patterns

Never build:

- Home screens
- dashboard dumps
- widget walls
- navigation-first experiences
- app launchers
- permanent data displays

Avoid:

`Here is everything.`

Prefer:

`Here is what matters.`

The more permanent information the default shell displays, the more JARVIS
drifts back toward generic software.

## Emotional Goal

The user should not feel:

- that they opened software
- that they need to search for context
- that they have to reassemble life state from dashboards

The user should feel:

- that JARVIS is here
- that JARVIS has been paying attention
- that continuity has been preserved
- that the system is ready to move with them immediately

This doctrine answers a different question than the product vision.

The product vision answers:

`What is JARVIS?`

This doctrine answers:

`What should it feel like to enter JARVIS?`
