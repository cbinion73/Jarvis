# JARVIS Rejoin Operation Milestone

## Purpose

This milestone exists to prove the JARVIS operating model.

Success is not measured by features completed.

Success is measured by whether Chris experiences:

`JARVIS feels like a living operation I rejoined.`

This is a proof plan.

It is not a roadmap.

It is not a doctrine document.

It is not a feature backlog.

It exists to define the smallest build slice that can validate the core
JARVIS interaction model.

## Success Scenario

Chris launches JARVIS.

JARVIS performs Operational Handoff.

Chris feels continuity restored.

Chris asks:

`Am I ready for Summer Camp?`

JARVIS materializes the Summer Camp Mission.

JARVIS enters Chief of Staff posture.

Chris asks:

`Should I rent a trailer?`

JARVIS transitions to Advisor posture.

Chris says:

`Let's do it.`

JARVIS transitions to Operator posture.

A next action is created.

No navigation occurs.

No applications are opened.

No dashboards are visited.

Everything emerges from conversation.

## Build Slice 1: Permanent Operating Layer

### Required

- Conversation Presence
- Continuity Spine
- Ambient Intelligence
- Mission Gravity

### Done when

JARVIS feels alive even with no workspace visible.

The permanent operating layer communicates:

- presence
- continuity
- readiness
- active operation

It does not communicate:

- dashboard information
- app structure
- screen hierarchy

## Build Slice 2: Operational Handoff

### Required

- Greeting
- What Changed
- What Matters
- Recommendation
- Invitation To Act

### Rules

- transient
- interruptible
- disappears when conversation begins

### Done when

JARVIS feels like an operation already in progress.

The user should feel that continuity is restored before work begins.

## Build Slice 3: Mission Object

### Only object supported

- Summer Camp Mission

### Constraints

- no additional domains
- no broad multi-object surface
- no generalized life graph UI yet

### Done when

The Summer Camp mission can materialize dynamically from conversation.

## Build Slice 4: Postures

### Only postures supported

- Chief of Staff
- Advisor
- Operator

### Constraints

- do not implement all six postures yet
- do not add Builder, Steward, or Companion representations in this milestone

### Done when

The same Summer Camp mission appears differently depending on posture.

At minimum:

- Chief of Staff reveals readiness, risks, and summary state
- Advisor reveals comparisons, tradeoffs, and recommendation
- Operator reveals next action, progress, and execution posture

## Build Slice 5: Intent Routing

### Only intent types supported

- Status Inquiry
- Decision Request
- Mission Update

### Done when

Conversation automatically selects the correct posture for the Summer Camp
mission and transitions naturally between them.

The user should not have to choose:

- a page
- a mode
- a section
- a mission tab

## Explicit Non-Goals

Do not build:

- Health
- Finance
- Publishing
- Faith
- Family
- Workshop
- Home Automation

These are future domains.

The model must be proven first.

Do not build:

- full agent visibility
- generalized app navigation
- dashboard substitutes
- multi-domain command center
- posture selectors
- complete life graph UI

## Completion Criteria

A user can:

1. Open JARVIS
2. Receive Operational Handoff
3. Ask about Summer Camp
4. See the mission materialize
5. Request a decision
6. See posture transition
7. Commit to action
8. See operator state emerge
9. Be unable to explain where the Summer Camp mission "lives" other than "I asked JARVIS about Summer Camp"

Without navigating anywhere.

If this experience works, the doctrine is validated.

## Evaluation Standard

The primary question is not:

`Did we ship enough features?`

The primary question is:

`Did this stop feeling like software and start feeling like a living operation?`

If the answer is no, more features will not save it.

If the answer is yes, the operating model is real and can be scaled across:

- health
- faith
- workshop
- home
- publishing
- finance
- family

and the rest of life domains later.

## Implementation Discipline

Ignore everything outside this milestone unless it is required to prove the
experience.

Do not optimize for breadth.

Do not optimize for feature count.

Do not optimize for parity with legacy surfaces.

Optimize only for one undeniable experience:

`Good afternoon Chris. While you were away...`

Then:

- the question is spoken
- the mission materializes
- the posture shifts
- the next action becomes real

That moment is the proof.
