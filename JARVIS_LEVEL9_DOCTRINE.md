# JARVIS Level 9 Doctrine

JARVIS is being built toward Level 9: an autonomous life operating system, not merely an app, chatbot, dashboard, or collection of disconnected features.

Level 9 means JARVIS can perceive context, maintain memory, coordinate specialized agents, plan work, execute approved actions, explain what it is doing, show what it knows, recover from failure, ask for consent when needed, protect secrets/private data, and operate within clear trust boundaries.

## Operating Doctrine

- Lanes organize work.
- Slices advance capability.
- Seams prove progress.
- Demos make progress tangible.
- Functions make substrate callable.
- Front-end surfaces make autonomy visible and controllable.

Work substrate-first, but never invisible-only. Every backend slice must produce something Chris can open, run, inspect, or test.

## For Every Slice

1. Preserve existing work.
2. Build the smallest meaningful Level 9 capability.
3. Expose it through a function, API, workflow, test, CLI, or UI surface.
4. Add tangible proof such as a debug console, local demo page, seeded workflow, activity timeline, memory inspector, approval queue, visible log, JSON output, or test report.
5. Run relevant checks.
6. Commit a clean seam.
7. Leave a progress note explaining what changed, why it matters, what Chris can test, checks run, commit created, blockers, and next slice.

Begin functions and front-end proof surfaces now, but only when tied to real substrate. Prefer trust/control surfaces over broad visual polish.

## Pause Conditions

Pause before destructive actions, overwriting prior work, deleting branches/worktrees, force-pushing, rebasing mature branches, irreversible migrations, secrets/private data exposure, external communications, purchases, paid services, production/live changes, or mature-live consent decisions.

Do not pause for safe local implementation, tests, documentation, local demos, mock data clearly marked as mock, or reversible code changes in trusted lanes.

## Quality Bar

Prefer small real progress, working vertical proof, observable autonomy, durable architecture, clean commits, and tangible test paths.
